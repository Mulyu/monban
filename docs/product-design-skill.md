# monban プロダクトデザインスキル

monban の設計・ドキュメント作成時に従うべき原則。

---

## 設計原則

### 言語非依存

- AST、import 解析、型情報など言語固有の機能に依存しない
- ファイルシステムの走査（glob / パス解析）とプレーンテキストの正規表現のみで動作する
- TypeScript / Ruby / Python / Go など、どの言語のプロジェクトでも同じルールが使える

### 検出のみ、修正はしない

- monban は違反の**検出**に集中する。`--fix` はサポートしない
- 修正はエージェントまたは人間が行う。monban はチェックのみ

### チェックの分類

| コマンド | 対象 | 判定手段 |
|---------|------|---------|
| `monban path` | パス構造（存在、命名、深度、数） | glob / パス文字列 |
| `monban content` | ファイル内容（禁止・必須パターン） | 正規表現 |
| `monban comment` | コメント率 | 行カウント |
| `monban size` | ファイル行数 | 行カウント |
| `monban doc` | ドキュメントとコードのハッシュ整合 | SHA256 |

### ルール設定の統一

- 全ルールのセレクタは `path`（glob パターン）で統一する
- `pattern` ではなく `path` を使う（「パターン」ではなく「場所」起点の思考）
- チェックの方向は一方向に絞る（例: naming は「場所→名前」のみ、「名前→場所」はやらない）

### 重複の排除

- 同じことを別の書き方で表現できるルールは統合する
- 例: extension_guard は forbidden の `path: "src/**/*.js"` で表現できるので独立ルールにしない
- 例: co_location は required の companions モードで表現できるので独立ルールにしない

---

## ドキュメント原則

### README

- 全コマンドの概要と設定例を記載する
- 各コマンドの詳細は `docs/` 配下の個別ファイルへリンクする
- コンセプト、インストール、エージェント統合の手順を含める

### docs/ 配下

- コマンドごとに 1 ファイル（`path.md`, `content.md` など）
- 各ファイルの構成:
  1. コマンドの概要と CLI 使用例
  2. ルール一覧表
  3. 設定全体の YAML 例
  4. ルールごとの詳細（目的、設定例、フィールド表、出力例）
  5. 共通出力フォーマット

### 設定例

- YAML の設定例は言語固有にしない。複数言語の例を混ぜて書く
- 出力例は実際の CLI 出力イメージを含める
